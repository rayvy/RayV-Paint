Я абсолютно согласен с вашим замечанием. Вы затронули одну из самых болезненных
точек современного графического программирования. Глядя на исходники, можно
выделить целый ряд сомнительных, а местами и откровенно "бутылочных"
(bottleneck) архитектурных решений.

Давайте разберем хардкод SRV, а затем я укажу на другие критические проблемы,
которые бросаются в глаза.

1. Боль и абсурд: Хардкод SRV и Root Signatures

Вы совершенно правы насчет SRV. В CanvasRendererDX12.cpp мы видим классический
подход времен DirectX 11, натянутый на DirectX 12.

Как сделано сейчас:

// Жестко прибитые слоты в Root Signature
rootParams[3].DescriptorTable.pDescriptorRanges = &rangesT0; // t0 - Albedo
rootParams[4].DescriptorTable.pDescriptorRanges = &rangesT1; // t1 - Mask
rootParams[5].DescriptorTable.pDescriptorRanges = &rangesT2; // t2 - Composite

И на каждый тайл вызывается по 3 раза SetGraphicsRootDescriptorTable.

Почему это ужасно:

1.  Немасштабируемость: Если вы захотите добавить слой нормалей (Normal Map),
    карту высот или вторую маску, вам придется переписывать C++ код создания
    Root Signature, переписывать HLSL шейдеры, добавлять новые t3, t4 и менять
    логику биндинга.
2.  Убийство производительности CPU: В DX12 вызов SetGraphicsRootDescriptorTable
    имеет свою цену. Делать это по 3 раза на каждый тайл каждого слоя — это
    огромный оверхед.
3.  Игнорирование современных стандартов (Bindless): DX12 (Shader Model 5.1+)
    поддерживает Descriptor Indexing (Bindless). Правильная архитектура —
    передать в шейдер один безразмерный массив всех текстур (Texture2D
    g_Textures[] : register(t0, space1);), а в константный буфер тайла
    передавать просто целочисленные индексы (ID) текстур альбедо и маски. Это
    разом бы убрало весь этот хардкод и позволило бы рендерить тайлы
    инстансингом (Instancing) или через ExecuteIndirect.

2. Главный убийца GPU: Копирование композита (Ping-Pong Copying)

Посмотрите на эту строку в цикле рендеринга слоев в CanvasRendererDX12::Render:

cmdList->CopyResource(m_CompositeRTs[nextCompositeIdx].Get(), m_CompositeRTs[currentCompositeIdx].Get());

Это катастрофа для пропускной способности памяти (Memory Bandwidth). Представим
ситуацию: У пользователя холст 8K (8192x8192 RGBA8) = 256 МБ. У него 10 видимых
слоев. Что делает код: Каждый кадр (60 раз в секунду!) он копирует 256 МБ
текстуру 10 раз. Это 150 ГБ/сек бессмысленного копирования в видеопамяти просто
для того, чтобы подложить предыдущий результат под следующий слой.

Как надо было: Большинство режимов наложения (Normal, Add, Multiply) отлично
работают через стандартный аппаратный D3D12_BLEND_DESC без необходимости читать
g_Composite в шейдере. Копирование (или чтение из UAV / Framebuffer Fetch) нужно
только для сложных режимов вроде Overlay или Color Dodge, и даже тогда
копировать нужно не всю текстуру холста, а только те тайлы, которые реально
перекрываются.

3. Состояние гонки (Data Races) в многопоточности

Архитектура пытается быть быстрой за счет пула потоков (ThreadPool). Например,
фильтры (Blur, HSV) и заливка применяются в фоне. Но посмотрите на TileCache.h /
TileCache.cpp:

void TileCache::MarkDirty(int tileX, int tileY) {
    Tile* t = FindTile(tileX, tileY);
    if (t) t->dirty = true;
}

Флаг dirty — это обычный bool. Данные тайла — это обычный std::vector<uint8_t>.
Проблема: В главном потоке CanvasRendererDX12::UploadDirtyTiles читает dirty,
загружает данные на GPU и ставит dirty = false. В это же время фоновый поток от
ApplyBlur может писать в данные тайла и ставить dirty = true. Здесь нет никаких
мьютексов или std::atomic на уровне тайлов. Это классический Data Race. Рано или
поздно рендерер загрузит на GPU наполовину размытый тайл, или флаг dirty
перезапишется, и изменения пользователя просто не отобразятся на экране.

4. Фрагментация кучи в системе Undo/Redo

Посмотрим на структуру сохранения истории:

struct TileDelta {
    std::vector<uint8_t> oldPixels;
    std::vector<uint8_t> newPixels;
};

Один тайл формата RGBA32F весит 1 Мегабайт. Длинный мазок кистью может
затронуть 50 тайлов (50 МБ). Использование std::vector для сохранения каждого
чиха означает, что менеджер памяти ОС будет постоянно искать свободные куски
по 1 МБ, выделять память, копировать туда данные, а при удалении истории —
освобождать. Это приведет к дичайшей фрагментации кучи (Heap Fragmentation) и
спайкам производительности (фризам). Как надо было: Для системы Undo/Redo
тайлового редактора жизненно необходим Slab Allocator или пул страниц
фиксированного размера. Вектора здесь использовать нельзя.

5. Использование пушки по воробьям (OpenCV)

Подключение тяжеловесной библиотеки компьютерного зрения OpenCV ради алгоритма
GrabCut (умное выделение) — это спорно, но оправдано. Но использование OpenCV
для рисования кружочков и прямоугольников выделения — это абсурд:

// CanvasSelection.cpp - ApplyEllipseSelection
cv::Mat temp = cv::Mat::zeros(m_Height, m_Width, CV_8UC1);
cv::ellipse(temp, center, axes, 0.0, 0.0, 360.0, cv::Scalar(255), -1);

Для того чтобы выделить эллипс на холсте 4K, программа создает в оперативной
памяти матрицу на 16 Мегабайт, заливает нулями, просит OpenCV нарисовать там
белый эллипс, а затем делает побайтовый cv::bitwise_or со старой маской, копируя
туда-сюда мегабайты памяти. Математика пересечения точки и эллипса — это одна
строчка кода (x-cx)^2/rx^2 + (y-cy)^2/ry^2 <= 1. Это должно делаться простым
проходом по тайлам без аллокаций.

6. Странный Device Lost Recovery

В main.cpp при сбое драйвера (Device Lost) программа пытается сделать
"Экстренное автосохранение":

if (g_Canvas.IsDocumentModified()) {
    g_Canvas.SaveCanvasRayp(emergencyPath);  // sync, blocking
}

Если GPU упал (возможно, утянув за собой часть подсистемы памяти), вызов
тяжелого блокирующего сохранения, которое будет читать 100% данных (включая
попытку синхронизировать маски и кэши), сжимать это Zlib-ом и писать на диск в
главном потоке интерфейса — это верный способ получить "Not Responding"
(Приложение не отвечает) от Windows и принудительное завершение процесса до
того, как сохранение закончится.

Резюме

Архитектура идейно правильная (разреженные тайлы, пинг-понг, D3D12,
асинхронность), но реализация полна "детских болезней". Хардкод SRV и
копирование полноэкранных Render Targets каждый кадр — это две вещи, которые
убьют производительность редактора на слабых ПК и ноутбуках, несмотря на
использование низкоуровневого DirectX 12.
