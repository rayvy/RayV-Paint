На основе предоставленного исходного кода подготовлен подробный технический отчет о текущей архитектуре приложения **RayVPaint** (высокопроизводительного растрового редактора для технических художников). 

Приложение построено на базе графического API **DirectX 12**, графического интерфейса **Dear ImGui**, библиотеки компьютерного зрения **OpenCV** и кроссплатформенной оконной библиотеки **GLFW**.

# Архитектурный отчет по проекту RayVPaint

## 1. Общий обзор архитектуры (High-Level Architecture)

Проект спроектирован с разделением на логические слои управления данными, интерактивного интерфейса и аппаратного ускорения. Взаимодействие компонентов можно представить следующей схемой:

```
[ Win32 / GLFW Window ] <--- События ввода, планшета (WM_POINTER), Drag & Drop
        |
        v
[ main.cpp (Оконный цикл) ]
        |
        +---> [ Dear ImGui (UI Panels) ] <--- Отображение логов, слоев, настроек кисти
        |
        +---> [ KeymapManager ] <--- Раскладко-независимые горячие клавиши
        |
        v
[ Canvas (Контекст документа) ] <--- Логика слоев, масок, выделений, трансформаций
        |
        +---> [ TileCache (CPU RAM) ] <--- Разреженное хранение пикселей (256x256)
        |
        +---> [ UndoRedoManager ] <--- Транзакционная история (дельта-тайлы)
        |
        v
[ CanvasRendererDX12 ] <--- Двухпроходный пинг-понг рендеринг слоев на GPU
        |
        +---> [ Dx12Device ] <--- Инициализация D3D12, Swapchain, дескрипторы
        +---> [ Dx12AsyncUploader ] <--- Асинхронная загрузка тайлов на COPY-очереди
        +---> [ GpuComputeTools ] <--- Вычисления на GPU (Flood Fill)
```

### Ключевые паттерны и концепции:
1. **Разреженное тайловое представление данных (Sparse Tiling)**: Изображение не хранится в виде непрерывного массива пикселей в памяти. Вместо этого оно разбито на сетку независимых тайлов размером $256 \times 256$. Память выделяется лениво — только под те участки изображения, на которых есть пиксели.
2. **Пинг-понг композиция на GPU (Ping-Pong Compositing)**: Рендерер объединяет слои в общую картину, последовательно переключаясь между двумя текстурами рендеринга (Render Targets) с использованием аппаратного смешивания и шейдеров blend-режимов.
3. **Асинхронность и многопоточность**: Задачи ввода-вывода (сжатие Zlib, генерация DDS), сложные операции выделения (OpenCV GrabCut) и загрузка текстур на видеокарту выполняются параллельно в фоновом режиме, не блокируя основной поток отрисовки.

## 2. Представление данных: Система разреженного тайлирования (`TileCache`)

Архитектурным фундаментом обработки изображений в редакторе является связка классов `TileCache` и `Layer`.

### `TileCache` (CPU-сторона):
* **Тайловая сетка**: Изображение разбито на блоки `TILE_SIZE = 256`. Вектор `m_Tiles` хранит структуру `Tile`, содержащую сырой буфер байт `data`, флаг загрязнения `dirty` для GPU-апдейтов и метку времени последнего доступа `lastAccess`.
* **Форматы пикселей (`CanvasPixelFormat`)**:
  * `RGBA8` (4 байта на пиксель) — для стандартного рисования, масок и текстур низкой точности. Размер тайла на CPU: **256 КБ**.
  * `RGBA32F` (16 байт на пиксель, IEEE Float) — для HDR-карт, карт высот (Heightmaps) и прецизионной графики. Размер тайла на CPU: **1 МБ**.
* **CPU LRU-кэширование (Least Recently Used)**: Ограничивает объем оперативной памяти, занимаемой открытыми документами. Если количество загруженных тайлов превышает `m_MaxTilesInRAM`, наиболее старые негрязные (чистые) тайлы автоматически выгружаются из оперативной памяти.

```
       +-----------------------------------+
       |     Canvas (Размер: 1024x1024)     |
       |  +-------------+---------------+  |
       |  |  Tile(0,0)  |   Tile(1,0)   |  |
       |  |  [256x256]  |   (Не создан/ |  |
       |  |  (Выделен)  |  прозрачный)  |  |
       |  +-------------+---------------+  |
       |  |  Tile(0,1)  |   Tile(1,1)   |  |
       |  | (Выгружен)  |   [256x256]   |  |
       |  |   на диск   |   (Выделен)   |  |
       |  +-------------+---------------+  |
       +-----------------------------------+
```

### `Layer` (Слои):
* Содержит имя, флаги видимости, прозрачность, режим наложения (`BlendMode`) и указатель на первичный `TileCache`.
* Реализует концепцию **неразрушающих фильтров** (`LayerFilter`): список эффектов (размытие,Curves, HSV-коррекция, инверсия альфы, шум), которые применяются "на лету" к оригинальным пикселям, сохраняя результат во временный кэш `filteredCache`.
* **Маска слоя (`mask`)**: Плоский массив байт (одноканальный R8), имеющий те же размеры, что и холст.

## 3. Графический конвейер: Система отрисовки и композиции (`CanvasRendererDX12`)

Рендерер разработан с упором на минимизацию накладных расходов процессора при отправке команд DirectX 12.

### Конвейер рендеринга кадра:
1. **Калибровка ресурсов композита**: При изменении размеров холста пересоздаются две ping-pong текстуры `m_CompositeRTs[2]` формата `DXGI_FORMAT_R8G8B8A8_UNORM` и дескрипторы для них.
2. **Инкрементальная загрузка тайлов**:
   * Для каждого активного `TileCache` слоев на GPU выделяется соответствующий `LayerGpuResources`.
   * Функция `UploadDirtyTiles` собирает только измененные тайлы (у которых `dirty == true`) и отправляет их на видеокарту.
3. **Проход композиции (Ping-Pong Blending)**:
   * Очищается начальный буфер `m_CompositeRTs[0]`.
   * Цикл по всем видимым слоям снизу вверх:
     * Предыдущий накопленный результат копируется из текущего композита в качестве подложки в следующий композитный буфер с помощью `cmdList->CopyResource`.
     * Устанавливается пайплайн `m_PsoLayerBlend` с включенным аппаратным смешиванием (`SrcBlend = SRC_ALPHA`, `DestBlend = INV_SRC_ALPHA`).
     * Для каждого выделенного GPU-тайла слоя обновляются константные буферы (позиция на экране, прозрачность, режим смешивания) и рисуется полигональная сетка из двух треугольников (6 вершин).
     * Пиксельный шейдер `PSTileBlend` (`CanvasTiles.hlsl`) принимает текстуру тайла, маску слоя, а также текстуру предыдущего композита. Внутри шейдера математически рассчитываются режимы наложения (Multiply, Screen, Overlay, Add, Subtract, Darken, Lighten, HardLight, SoftLight) и выводится итоговый цвет.
4. **Вывод на экран (Viewport Pass)**:
   * Пайплайн переключается на `m_PsoCheckerboard` (`Canvas.hlsl`).
   * Итоговый композитный буфер отрисовывается во фреймбуфер вьюпорта ImGui. На этом этапе пиксельный шейдер генерирует динамическую шахматную сетку (checkerboard) для отображения прозрачных областей, применяет маски каналов (R, G, B, A) и рисует серую рамку вокруг границ холста.
5. **Отрисовка контура выделения**: Если на холсте есть активная маска выделения, пайплайн `m_PsoSelectionOutline` считывает текстуру маски выделения и в местах резкого перепада яркости (границы) рисует контрастные движущиеся штрихи ("муравьиную дорожку") с использованием встроенной функции `u_ViewportFlags.z` (время).

### Оптимизации DX12:
* **Кольцевой буфер констант (Constant Buffer Ring)**: Выделен единый буфер `m_ConstantBufferUpload` размером 32 МБ. Данные параметров отрисовки слоев и тайлов записываются туда последовательно с выравниванием на 256 байт, минимизируя вызовы маппинга памяти и связывания дескрипторов (`SetGraphicsRootConstantBufferView`).
* **Сборка мусора и обрезка лимитов**: Функция `GarbageCollectGpuLayers` освобождает дескрипторы и текстуры тайлов GPU, если соответствующие слои были удалены на CPU. Функция `TrimGpuTileCache` следит за лимитом `m_MaxGpuTiles` и выгружает старые тайлы из видеопамяти при достижении порога.

## 4. Конвейер интерактивной обработки и рисования

Приложение реализует быстрый процессинг за счет локальной обработки данных на CPU и GPU.

### Движок рисования (`PaintEngine` на CPU):
* **Оптимизированный штамп (Stamp-based Painting)**: Кисть рисует непрерывную линию путем последовательного наложения круглых штампов (`DrawStrokeSegment`). Расстояние между штампами контролируется параметром `spacing`.
* **Локализация по тайлам**: Вместо обхода всего изображения рассчитывается ограничивающая рамка кисти (Bounding Box). На основе этих координат вычисляются только те тайлы `TileCache`, которые пересекаются с кистью. Вызывается `cache.LockTile(tx, ty)`, и пиксели изменяются непосредственно в памяти конкретного тайла.
* **Функции кисти**: Стирание (Eraser), жесткость краев (Hardness), маскирование по выделению, а также координатная симметрия кисти по горизонтали и вертикали.

### Масштабируемые операции над пикселями (`CanvasAdjustments`):
* Операции деструктивной коррекции (InvertAlpha, ApplyBlur, ApplyHSV, ApplyCurves, ApplyNoise) распараллелены на пуле потоков процессора (`ThreadPool`). 
* Вся область изображения обрабатывается блоками в рабочих потоках. При этом изменения вносятся только в те пиксели, вес которых в маске выделения `m_SelectionMask` больше нуля.

```cpp
// Пример многопоточного обхода тайлов в CanvasAdjustments.cpp
std::vector<std::future<void>> futures;
for (int w = 0; w < numWorkers; ++w) {
    futures.push_back(ThreadPool::Get().Enqueue([&]() {
        size_t idx;
        while ((idx = nextTileIndex.fetch_add(1)) < totalTiles) {
            // Локальная обработка пикселей внутри тайла activeTiles[idx]
        }
    }));
}
for (auto& f : futures) f.wait();
```

### Трансформация "Move Pixels" (Перемещение пикселей):
* При активации инструмента область, попавшая в выделение, вырезается из исходного слоя и помещается во временный тайловый кэш `m_FloatingTileCache`.
* Пользователь может интерактивно перемещать, масштабировать и вращать вырезанные пиксели с помощью экранного манипулятора (Gizmo).
* На этапе композиции `CanvasRendererDX12` накладывает плавающие пиксели поверх слоя, вычисляя новые координаты пикселей в вершинном шейдере `VSTileMain` с использованием матрицы аффинных преобразований (перенос, поворот вокруг центра `centerX`/`centerY`, масштаб).
* При подтверждении трансформации (`CommitMovePixels`) пиксели перепроецируются обратно на сетку тайлов основного слоя с билинейной фильтрацией.

## 5. Выделение областей и компьютерное зрение

Редактор предоставляет продвинутые инструменты выделения пикселей, сочетающие CPU и GPU методы:

### Волшебная палочка (`MagicWand`):
* **Contiguous (Связанная область)**: Потоковая реализация BFS (обход в ширину) `MagicWandBFS_Tiled` оптимизирована под тайловую структуру. Имеется локальное кэширование указателя на последний запрошенный тайл для ускорения обращений по координатам `(x, y)`.
* **Global Threshold (Несвязанная область)**: Многопоточная фильтрация `MagicWandGlobalThreshold` проверяет соответствие цвета во всех тайлах по евклидову расстоянию в цветовом пространстве RGB с заданной погрешностью (`tolerance`).
* **GPU-ускорение (Flood Fill)**: Модуль `GpuComputeTools` реализует альтернативный быстрый алгоритм заливки на вычислительных шейдерах (`FloodFill.hlsl`). Вызывается инициализация маски (`CSInit`), после чего шейдер распространения (`CSMain`) запускается итеративно, выполняя волновое заполнение маски непосредственно в памяти видеокарты.

### Умное выделение (`SmartSelect`):
* Интегрирует классический алгоритм сегментации **GrabCut** из библиотеки **OpenCV**.
* Пользователь рисует лассо вокруг объекта. Программа рассчитывает ROI (область интереса) с запасом в 15 пикселей, копирует этот фрагмент в матрицу `cv::Mat`, а затем запускает фоновый рабочий поток для выполнения итерационного разделения планов `cv::grabCut` на передний (`cv::GC_PR_FGD`) и задний (`cv::GC_BGD`) фон. По завершении маска ROI проецируется обратно на общую маску выделения.

## 6. Отказоустойчивость и транзакционная история (Undo/Redo)

Система истории изменений разработана с жесткими рамками экономии памяти (Task 12).

* **Транзакционная дельта-история**: Программа не сохраняет копии слоев целиком на каждом шаге. Вместо этого в `BackupTile` перед любой операцией модификации оригинальное состояние тайла сохраняется в структуру `TileDelta`.
* **Тайловый снимок (`TileDelta`)**:
  * Идентификатор слоя и координаты тайла `(tileX, tileY)`.
  * `oldPixels` — снимок сырых байт до транзакции (размером 256 КБ для RGBA8).
  * `newPixels` — снимок сырых байт по окончании транзакции.
* **Менеджер `UndoRedoManager`**:
  * Хранит стек команд в виде списка умных указателей `std::shared_ptr<UndoCommand>`.
  * Реализует метод `EnforceLimits()`, который при превышении максимального числа шагов (`max_undo_steps`) или лимита памяти в мегабайтах (`max_undo_memory_mb`) удаляет старые транзакции из стека, освобождая занимаемую ими системную память.

## 7. Файловый ввод-вывод и работа с форматами

### Конвертер DDS (`DdsHelper`):
* Чтение сжатых текстур реализовано с помощью встраиваемой библиотеки `bcdec.h`. Приложение напрямую декодирует блоки алгоритмов **BC1, BC2, BC3, BC4, BC5 и BC7** в несжатый формат и импортирует их построчно в `TileCache`.
* Запись сжатых текстур реализована через утилиту **DirectXTex `texconv.exe`**. Слой экспортируется во временный uncompressed DDS-файл, после чего через вызов системного процесса Windows `CreateProcessW` запускается `texconv.exe` с соответствующими флагами (генерация мип-уровней, фильтрация Cubic/Fant/Linear, скорость сжатия и формат назначения).

### Пакетный формат проектов (`.rayp`):
Собственный бинарный архив проекта, упаковывающий все слои и настройки. Структура файла:
1. **Заголовок-магическое число**: Строка `"RAYP"` (4 байта).
2. **Версия формата**: `uint32_t` (4 байта).
3. **Размер метаданных**: `uint64_t` (8 байт).
4. **JSON-метаданные**: Текст в кодировке UTF-8, содержащий размеры холста, структуру слоев (имена, видимость, прозрачность) и параметры экспорта.
5. **Данные пикселей слоев**: Последовательные блоки упакованных данных для каждого слоя. Каждый блок предваряется размерами `uncompressedSize` (8 байт) и `compressedSize` (8 байт), после чего следуют пиксели, сжатые с помощью алгоритма **zlib** (`stbi_zlib_compress` на уровне сжатия 8).

## 8. Инициализация и системная интеграция

* **Высокоточное перьевое рисование**: Для поддержки графических планшетов в `main.cpp` используется переопределение стандартной оконной процедуры Windows (`GWLP_WNDPROC`). Программа обрабатывает системные события Win32 API для считывания силы нажатия пера (`g_PenPressure`), которая затем динамически модулирует радиус, жесткость и непрозрачность кисти.
* **Обработка потери устройства (Device Lost Recovery)**: При сбое видеокарты или обновлении драйверов (`DXGI_ERROR_DEVICE_REMOVED`), программа перехватывает ошибку, осуществляет экстренное автосохранение проекта в резервную копию `emergency_autosave.rayp` на жесткий диск, полностью сбрасывает все GPU-ресурсы, воссоздает устройство D3D12 и повторно инициализирует рендерер, предотвращая потерю прогресса пользователя.
* **Раскладко-независимые горячие клавиши**: `KeymapManager` регистрирует действия пользователя по физическим скан-кодам клавиш (`glfwGetKeyScancode`), благодаря чему сочетания клавиш работают корректно независимо от текущей активной раскладки клавиатуры в ОС.
* **Интеграция с системным буфером обмена**: Модуль `ClipboardHelper` использует функции Win32 API `OpenClipboard`, `SetClipboardData` и формат `CF_DIB` для копирования и вставки изображений между RayVPaint и другими графическими редакторами (например, Photoshop), налету конвертируя форматы пикселей и выравнивание строк.
