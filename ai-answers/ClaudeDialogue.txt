Claude:
Короткий ответ: **современного API из §9 недостаточно для обоих сценариев** — не потому что чего-то фундаментально не хватает в pixel-доступе, а потому что оба сценария по своей природе **асинхронные и внешние**, а API сейчас спроектирован как синхронный, "всё происходит внутри одного `on_ui()`-кадра".

Разберу по обоим кейсам отдельно, с привязкой к конкретным местам доки.

## Что уже закрыто

- **Зона/выделение**: `has_selection()/get_selection()` — маска `w*h` есть, можно вычислить bbox и вырезать регион.
- **Чтение/запись региона**: `get_pixels/set_pixels`, `get_layer_rgba8/set_layer_rgba8` — годится и для отправки init-image в img2img, и для заливки результата обратно.
- **UI для промпта/попапа**: `input_text`, `combo` (выбор модели из списка), `checkbox`, `button` — форма "промпт + модель + Generate" собирается уже сейчас.

Т.е. "статическая" часть интерфейса плагина — есть. Проблема начинается там, где должна начаться сеть/долгая генерация.

## Чего категорически не хватает — под ComfyUI-плагин

1. **Асинхронность / потокобезопасность не задокументирована.** `on_ui()` — единственный recurring-хук, вызывается каждый ImGui-кадр. Генерация в ComfyUI — это 5–60 секунд. Синхронный HTTP-вызов внутри `on_ui()` = замороженный UI-поток на всё это время. Документация нигде не говорит, можно ли безопасно дёргать `rayv.doc.*` из воркер-треда — а без этой гарантии единственный легальный паттерн — "запустил тред, копишь результат в очередь, читаешь из `on_ui()` на каждом кадре, а `rayv.doc.set_*` вызываешь только из главного потока". Это рабочий паттерн, но он **не подтверждён контрактом**, а не потому что кода нет.
2. **Нет декодера изображений в `rayv`.** ComfyUI отдаёт PNG. Ни `PIL`, ни `numpy`, ни какой-либо `rayv.image.decode(...)` в доке не упомянуты. Если в embedded-интерпретаторе нет сторонних пакетов — писать PNG-парсер на чистом stdlib ради каждого such-плагина абсурдно. Нужно либо подтвердить набор бандленных пакетов, либо дать тонкий `rayv.image.decode_png(bytes) -> (w,h,rgba8_bytes)`.
3. **Undo не подключён.** §9.4 прямо: *"direct pixel writes are not on the undo stack"*. Генеративная заливка — это ровно то действие, где юзер ожидает Ctrl+Z как для обычного мазка. Нужен хотя бы `begin_undo_group()/end_undo_group()` вокруг `set_layer_rgba8`, иначе AI-generate ощущается как хак, а не как штатный инструмент.
4. **Нет preview/progress виджетов.** `rayv.ui` сейчас — это text/button/checkbox/input/combo. Нет `image()` (показать превью результата перед "Apply") и нет `progress_bar()`/спиннера. Для AI-workflow "сгенерировал → посмотрел → применил или отменил" это почти обязательный UX-элемент.
5. **`assets::AssetManager` не проброшен в Python вообще.** В §4 он есть как внутренняя runtime-система, но в §9 (Python API) про него ни слова. Если хочешь не просто "залить в текущий слой", а сохранить сгенерированную текстуру как переиспользуемый `user:`/`proj:` asset — из скрипта сейчас этого не сделать.

## Что не хватает — под Blender-мост

Тут отдельный нюанс: судя по тому, что реальный Blender bridge у тебя уже спроектирован на уровне Named Pipe + memory-mapped file с dirty-tile reuse — это, скорее всего, **нативная C++ подсистема**, а не Python-плагин поверх `rayv`. Так что вопрос стоит уточнить: ты хочешь мост *как нативный компонент* (тогда §9 вообще не при чём, это отдельная разработка) или *как user-plugin на Python* поверх текущего scripting API? Если второе:

1. **Нет события "тайл стал грязным".** `tile_size()`/`active_tile_count()` дают только текущее состояние, но нет `on_tiles_dirty(indices)` колбэка. Без него синхронизация из Python — это либо poll всего layer каждый кадр (дорого при 4K/8K), либо сам dirty-tile-reuse смысл теряется именно на границе "движок ↔ скрипт".
2. **Нет способа зарегистрировать новый operator** — плагин может только `invoke()` существующие id из `ActionCatalog`, но не добавить свой в `OperatorRegistry`. Для полноценного bridge-инструмента (со своим hotkey/меню-записью, а не только модальным окном) это ограничение.
3. Тот же пункт про потокобезопасность — если мост слушает pipe в фоне, нужна гарантия, что доставка данных в `rayv.doc` из не-main-потока безопасна, либо документированный способ передать управление в main thread.

## Если резюмировать в порядке приоритета для расширения API

1. **Контракт по потокам** (можно ли вызывать `rayv.doc.*` вне `on_ui()`, и как правильно передавать результат из воркера) — без этого пункта остальное не имеет смысла обсуждать.
2. **`rayv.image` decode/encode helper** (или подтверждение бандленных пакетов).
3. **Undo group API** вокруг pixel-write вызовов.
4. **`rayv.ui.image()` + `progress_bar()`** для preview/progress.
5. **Dirty-tile events** — нужно именно для bridge-сценария, для AI-плагина не критично.
6. **`rayv.assets` биндинг** — nice to have, не блокер для MVP.

Пункты 1–4 — это то, что превращает "можно вписать пиксели в холст" в "нормальный генеративный инструмент". Без них технически ComfyUI-плагин **можно** собрать (через ручной threading без гарантий), но это будет держаться на недокументированном поведении хоста, а не на контракте API.
